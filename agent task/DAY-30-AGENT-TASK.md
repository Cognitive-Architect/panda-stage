# Panda Stage Agent Task — Day 30

> **工单编号**：B-30/45  
> **角色**：Engineer  
> **来源**：`DAILY_PLAN.md` Day 30  
> **分支建议**：`chore/day-30-gate-b`  
> **任务类型**：端到端验收 + 编辑闭环 + Gate B  
> **唯一目标**：从空项目开始，在不修改代码的前提下完成约 30 秒内部 Alpha，验证素材、角色、镜头、画布、动作、对白、字幕、音频、嘴巴、撤销和项目重开形成完整编辑闭环。

## 【模块1】饱和攻击头部（通用增强版）

- **火力配置**：1 Agent（Engineer）
- **任务名称**：Gate B — No-Code 30-Second Internal Alpha
- **轰炸目标**：使用现有产品 UI 从零制作一个 5 镜头、2 角色、6～8 段音频、至少 5 个动作的约 30 秒项目，保存、关闭、重开并再次完整预览。
- **任务性质**：端到端验收 + 生产流程验证 + 止损决策
- **输入基线**：M3 Gate PASS；Day 26～29 时间轴、求值器、对白和多镜头预览全部完成；测试素材必须原创、项目自制或许可明确。
- **输出要求**：真实项目目录 + 30 秒预览证据 + 全过程耗时与失败次数 + `docs/test-receipts/GATE-B.md` + 唯一 PASS/FAIL 结论。
- **通用铁律**：
  1. 制作过程中不得编辑源代码、项目 JSON 或使用开发者控制台修画面。
  2. Gate 结论只能是 `PASS` 或 `FAIL`，禁止“基本可用”“差不多”。
  3. 任何“必须改代码才能改画面”的 P0 阻塞都直接判 FAIL，并冻结 M5。
  4. 所有耗时、失败次数、撤销次数和漂移结论必须来自真实记录。
  5. 不为过 Gate 临时降低 Day 1～29 已定义的验收标准。

## 【模块3】工单矩阵

### 基础信息

- **工单编号**：B-30/45
- **角色**：Engineer
- **依赖关系**：依赖 M3 PASS 与 Day 26～29；Gate B FAIL 时冻结 Day 31～45 实际开发。

### 输出交付物

- **必须产出**：
  - 一个从空项目制作的真实项目目录；
  - 素材来源与许可清单；
  - 约 30 秒项目预览录屏；
  - 保存关闭重开后的第二次预览证据；
  - 制作步骤时间记录；
  - 失败次数与绕路记录；
  - 至少 20 步 undo/redo 记录；
  - `docs/test-receipts/GATE-B.md`；
  - FAIL 时 `docs/decisions/M4-FAILURE-REPORT.md`。
- **制作清单**：
  1. 从空项目开始；
  2. 导入至少 1 个背景；
  3. 创建 2 个角色，每个至少 2 个表情；
  4. 导入 6～8 段外部音频；
  5. 创建 5 个镜头；
  6. 每个镜头完成基本摆位；
  7. 全项目至少使用 5 个动作预设；
  8. 添加对白、字幕和嘴巴动画；
  9. 完整预览约 30 秒；
  10. 保存、关闭、重开；
  11. 再次完整预览并对比；
  12. 执行至少 20 步撤销与重做边界验证。
- **必须包含**：
  - 全流程不改代码；
  - 不手改 project.json；
  - 5 镜头顺序、名称、时长重开一致；
  - 图层位置、变换和层级重开一致；
  - 动作事件重开一致；
  - 字幕、音频和嘴巴按时间工作；
  - 镜头边界无严重残影；
  - 暂停、跳转和停止无叠播；
  - 20 步撤销可用；
  - 所有失败与未验证项如实记录；
  - 结论唯一为 PASS 或 FAIL。
- **禁止包含**：
  - 为通过 Gate 手改代码或 JSON；
  - 只使用现成 demo 项目代替从空项目；
  - 把失败步骤从回执中删掉；
  - 只录最后结果、不记录制作过程；
  - 未重开项目就判一致；
  - Gate FAIL 后继续 M5；
  - 在本日顺手实现导出、混音或新功能。
- **交付证明**：
  - 项目目录与 `project.json`；
  - 素材清单；
  - 两次完整预览；
  - 逐步耗时表；
  - 失败/绕路表；
  - 自动化命令输出；
  - Gate B 回执与结论。

### 规模与复杂度观察

- 本日原则是“用软件做片”，不是“继续开发软件”。
- 只允许修复阻塞 Gate 的最小 P0；任何修复必须先写复现、再补回归测试，并在回执中单独列出。
- 若同一阻塞需要大重构，直接判 FAIL，创建 failure report，不得边做项目边偷偷重建系统。
- 记录耗时按固定步骤分类，避免最后只写模糊总时长。

### 自动化质量闸门

## 【模块3-B】地狱红线

1. M3 Gate 非 PASS 仍执行 Gate B → 停止。
2. 制作过程中编辑代码 → Gate FAIL。
3. 手改 project.json 绕过 UI → Gate FAIL。
4. 使用现成完整项目代替从空项目 → Gate FAIL。
5. 保存重开后内容不一致仍判 PASS → 严重违规。
6. 字幕、音频、动作或嘴巴明显错误仍判 PASS → Gate FAIL。
7. 未验证 20 步撤销仍判 PASS → Gate FAIL。
8. 删除失败/绕路记录以美化结果 → 严重违规。
9. Gate FAIL 后继续 Day 31 → 严重违规。
10. 自动化门禁失败仍判 PASS → Gate FAIL。
